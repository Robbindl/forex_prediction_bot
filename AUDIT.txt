But with C-07 (guard never called) and this reset, even if C-07 were fixed: after trade 1 closes at -1.8%, the new guard initializes at the reduced balance. The 5% check on trade 2 is against an already-smaller number, making the limit progressively looser.
Fix:

    with self._lock:
        self.account_balance = new_balance
        self._sizer.account_balance = new_balance
    # DO NOT reset DailyLossGuard — it resets daily via reset_daily()

H-08 — OilPrice API Returns Fake 1-Row OHLCV for CL=F

H-09 — Partial Take-Profit Levels Don't Reduce Position Size
File: execution/paper_trader.py:_check_exit():125
Severity: HIGH
Layer 7 creates 3-tier TP levels [0.5×, 1.0×, 1.5×]. When TP1 is hit, pos["tp_hit"] increments but the position size is unchanged. In real trading, partial TP means selling a fraction. Here the trade continues at full size until TP3 (1.5× the original TP), which is 50% farther than the strategy intended as its take-profit target. This systematically distorts realized P&L and win rates.
Fix:

pos["position_size"] -= partial_close_size
pnl_partial = PositionSizer.pnl(..., size=partial_close_size, ...)
# fire on_trade_closed with partial trade

## SECTION 6: ARCHITECTURAL PROBLEMS

### Problem 1: Three Separate Backtesting Systems
`backtest/engine.py`, `strategy_lab/backtest_engine_v2.py`, and `core/pipeline_reporter.py._run_fresh_backtest()` each implement backtesting independently with different risk models, different context handling, different Sharpe annualisation, and no shared logic. Results are not comparable across systems.

### Problem 2: Module-Level Side Effects
Eleven modules execute heavyweight operations at import time (DB queries, thread spawning, Redis connections, file writes). This makes testing impossible without `conftest.py` mocking, makes import order significant, and makes startup timing unpredictable.

### Problem 3: Two Telegram Bots With Overlapping Responsibilities
Bot 1 (commander) and Bot 2 (intelligence) share some functionality but are siloed. Health alerts are routed to Bot 2 but Bot 2's Telegram token is optional. When Bot 2 is absent, health monitoring is completely silent. There is no fallback to Bot 1 for health alerts.

### Problem 4: Redis as a Fragile Single Point of Failure
The entire architecture depends on Redis for signal propagation, dashboard updates, order flow, Phase 7 alerts, and intelligence events. Multiple modules permanently disable their Redis publishers on first error (`self._pub = None`) with no recovery. A brief Redis outage causes permanent silence in 8+ subsystems until bot restart.

### Problem 5: Wrong Account Size Hardcoded Everywhere
The `.env.example`, `Procfile`, `docker-compose.yml`, and `bot.py` all reference `--balance 30`. But `risk/position_sizer.py` uses specs for $10,000+. `PerformanceAnalyzer` uses `$10,000` as denominator. The system was designed for a larger account and the balance was changed without updating the dependent systems.
